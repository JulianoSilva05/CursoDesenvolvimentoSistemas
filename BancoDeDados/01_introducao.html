A Era dos Dados

Vivemos em um mundo movido a dados. Eles são o "novo petróleo".

Volume Explosivo

Geramos 2.5 quintilhões de bytes de dados por dia (IoT, Redes Sociais, Sensores).

Valor Estratégico

Empresas usam dados para entender clientes, prever tendências e tomar decisões.

Mas como organizar tudo isso de forma eficiente?

Antes dos Bancos de Dados

Imagine uma empresa guardando tudo em arquivos de texto ou planilhas soltas (Sistema de Arquivos).

Problema 1: Redundância
O mesmo dado (ex: endereço do cliente) repetido em vários arquivos. Se mudar, tem que atualizar em todos.
Problema 2: Inconsistência
O arquivo de "Vendas" diz que o cliente mora na Rua A, o de "Entregas" diz Rua B. Em quem confiar?
Problema 3: Isolamento e Segurança
Difícil cruzar dados de arquivos diferentes. Difícil controlar quem pode ler ou editar cada linha.

Evolução dos Bancos de Dados

  • Anos 60: Modelos Hierárquicos (Árvore) e em Rede (Grafo). Complexos e rígidos.
  • Anos 70 (A Revolução): Edgar F. Codd propõe o Modelo Relacional. Baseado em tabelas e matemática (álgebra relacional).
  • Anos 80: Padronização da linguagem SQL. Surgimento do Oracle, DB2.
  • Anos 90/00: BDs Orientados a Objeto e Open Source (MySQL, PostgreSQL).
  • Anos 2010+: NoSQL (Not Only SQL) para Big Data e flexibilidade (MongoDB, Cassandra).
  • Atualmente: NewSQL, BDs na Nuvem (DBaaS), Vector DBs (para IA).

O Modelo Relacional

Ainda é o mais utilizado no mundo corporativo. Baseia-se em Tabelas (Relações).

TABELA: Clientes +----+--------------+-------------------+ | ID | Nome | Email | <-- Colunas (Atributos) +----+--------------+-------------------+ | 1 | Ana Silva | ana@email.com | <-- Linha (Tupla/Registro) | 2 | João Souza | joao@email.com | +----+--------------+-------------------+

Entidade: O objeto do mundo real (ex: Cliente).
Atributo: As características (ex: Nome, Email).
Tupla: Uma instância única (ex: A Ana Silva).

Chaves: O Segredo das Relações

Como garantimos que cada registro é único e como ligamos tabelas?

Chave Primária (PK) - Primary Key

A Chave Primária é a coluna (ou conjunto de colunas) que identifica de forma EXCLUSIVA cada linha na tabela.

  • Unicidade: Não podem existir dois registros com a mesma PK.
  • Obrigatória: Todo registro deve ter uma PK e ela não pode ser vazia (NOT NULL).
  • Identidade: É o "RG" do registro no banco de dados.

Ex: CPF, Matrícula, Código do Produto.

Chave Estrangeira (FK) - Foreign Key

A Chave Estrangeira é o campo que estabelece o RELACIONAMENTO entre duas tabelas.

  • Conexão: Ela guarda o valor da PK de outra tabela.
  • Integridade: Garante que o vínculo seja válido (não posso vender para um cliente que não existe).
  • Navegação: Permite cruzar dados de tabelas diferentes.

Ex: O campo "id_cliente" na tabela de "Pedidos" aponta para o "id" do "Cliente".

SQL: A Linguagem Universal

É a linguagem usada para conversar com bancos relacionais.

-- DDL (Data Definition Language): Criando a estrutura CREATE TABLE clientes ( id INT PRIMARY KEY, -- Inteiro e Chave Primária nome VARCHAR(100), -- Texto até 100 caracteres email VARCHAR(100) ); -- DML (Data Manipulation Language): Manipulando dados INSERT INTO clientes (id, nome, email) VALUES (1, 'Ana Silva', 'ana@email.com'); -- DQL (Data Query Language): Consultando SELECT * FROM clientes WHERE id = 1;

O código acima cria uma "gaveta", guarda uma "pasta" e depois busca essa pasta.

NoSQL: Flexibilidade Moderna

Surgiu quando o modelo relacional (rígido) não aguentou a escala da Web (Google, Amazon, Facebook).

Exemplo: Modelo de Documentos (JSON)

{ "_id": 1, "nome": "Ana Silva", "contatos": { // Estrutura aninhada (difícil em tabelas) "email": "ana@email.com", "twitter": "@ana_dev" }, "interesses": ["Java", "SQL", "Docker"] // Lista flexível }

SGBDs Famosos: MongoDB (Documento), Redis (Chave-Valor), Cassandra (Colunar).

Modelagem de Dados (MER)

Antes de criar tabelas, desenhamos o Modelo Entidade-Relacionamento.

Retângulo

Representa a Entidade (Ex: Aluno)

Elipse

Representa o Atributo (Ex: Nome)

Representa o Relacionamento (Losango)

Exemplo: Aluno (Entidade) assiste (Relacionamento) Aula (Entidade).

Exemplo Prático de MER: Médico e Paciente

Vamos analisar um diagrama real para entender como representamos os dados graficamente.

1. Entidades (Retângulos)

Médico e Paciente. São os objetos principais do nosso sistema.

2. Relacionamento (Losango)

Atendimento. Conecta o Médico ao Paciente. Observe que o próprio atendimento tem um atributo: Data (quando ocorreu).

3. Atributos (Elipses)

  • Identificadores (Sublinhados): CRM (Médico) e CPF (Paciente). São chaves únicas que não se repetem.
  • Simples: Nome. Armazena um valor único.
  • Multivalorado (Elipse Dupla): Telefone. Indica que um paciente pode ter mais de um número.
  • Composto (Ramificações): Endereço. Para ser completo, ele precisa de Rua, Número e Bairro.
