Por que Modelar Dados?

Construir um banco sem modelar é como construir um prédio sem planta.

Comunicação

O modelo serve para alinhar o entendimento entre o Cliente (que conhece o negócio) e o Desenvolvedor (que conhece a técnica).

Prevenção de Erros

Corrigir um erro no papel (modelo) custa R$ 1. Corrigir depois que o sistema está pronto custa R$ 100.

Os 3 Níveis de Abstração

A modelagem acontece em etapas, do mais abstrato ao mais detalhado.

1. Modelo Conceitual (O QUE?)
Focado no negócio. Independente de tecnologia. Usa retângulos e losangos (MER).
Público: Cliente e Analista.
2. Modelo Lógico (COMO?)
Define a estrutura. Introduz Chaves Primárias (PK) e Estrangeiras (FK). Normalização.
Público: Desenvolvedor e Arquiteto.
3. Modelo Físico (ONDE?)
O código SQL específico para o SGBD (Oracle, MySQL, SQL Server). Tipos de dados exatos (VARCHAR(50), INT).
Público: DBA e SGBD.

Modelo Entidade-Relacionamento (MER)

Criado por Peter Chen. É a base de tudo.

ENTIDADE

Retângulo

Objeto do mundo real (Cliente, Produto).

ATRIBUTO

Elipse

Característica (Nome, Preço).

REL

Relacionamento (Losango): A ação que liga entidades (ex: Compra, Pertence).

Tipos de Atributos

Nem todo atributo é igual.

  • Simples (Atômico): Indivisível. Ex: Sexo (M/F).
  • Composto: Pode ser dividido. Ex: Endereço (Rua, Bairro, CEP).
  • Multivalorado: Pode ter vários valores. Ex: Telefone (Celular, Fixo, Trabalho). Representado com elipse dupla.
  • Derivado: Calculado a partir de outro. Ex: Idade (calculado da Data de Nascimento). Representado com elipse tracejada.
  • Identificador (Chave): Identifica a entidade. Ex: CPF. Atributo sublinhado.

Cardinalidade: A Regra do Jogo

Define quantas ocorrências de uma entidade se relacionam com a outra.

1:1 (Um para Um)
Um marido tem uma esposa (na lei brasileira). Um funcionário tem um CPF.
1:N (Um para Muitos) - O MAIS COMUM
Um cliente faz vários pedidos. Um pedido pertence a um cliente.
N:N (Muitos para Muitos)
Um aluno cursa várias disciplinas. Uma disciplina tem vários alunos.
(Gera uma tabela associativa no modelo lógico).

Entidades Fortes e Fracas

Entidade Forte

Existe independente de qualquer outra.

Ex: Cliente. Se deletar todos os pedidos, o cliente ainda existe.

Entidade Fraca

Sua existência depende de uma entidade forte.

Ex: Dependente (Filho). Se o Funcionário for demitido/excluído, o Dependente perde o sentido no sistema.

Transformando em Tabelas (Modelo Lógico)

Quando passamos para o modelo lógico, as regras mudam um pouco:

  • Entidades viram Tabelas.
  • Atributos viram Colunas.
  • Relacionamentos 1:N viram Chave Estrangeira (FK) no lado N.
  • Relacionamentos N:N viram uma Nova Tabela (Associativa).
Exemplo N:N (Aluno <-> Disciplina): Tabela Alunos (id, nome) Tabela Disciplinas (id, nome) Tabela Matriculas (id_aluno, id_disciplina, data) <-- Associativa
