O Mundo Relacional (SQL)

Dominante há 40 anos. Baseado em Tabelas e Esquemas Rígidos.

Características

  • Dados estruturados (linhas/colunas).
  • Esquema definido antes de inserir dados.
  • Relacionamentos fortes (FK).

Onde brilha?

  • Sistemas Financeiros (Bancos).
  • ERP / CRM.
  • Onde a integridade é vital.

ACID: A Garantia de Segurança

Todo banco relacional sério segue estas 4 regras para transações:

A - Atomicidade: Tudo ou nada. Se uma parte da transação falhar, tudo é desfeito (Rollback).
C - Consistência: O banco vai de um estado válido para outro estado válido.
I - Isolamento: Uma transação não interfere na outra enquanto acontece.
D - Durabilidade: Se o banco disse "salvo", está salvo no disco, mesmo que a energia caia.

O Mundo NoSQL (Not Only SQL)

Surgiu nos anos 2000 com a Web 2.0 (Google, Amazon, Facebook). O Relacional não aguentava o volume.

Características

  • Schema-less (Sem esquema fixo).
  • Alta performance de leitura/escrita.
  • Escalabilidade Horizontal fácil.

Onde brilha?

  • Big Data / IoT.
  • Redes Sociais.
  • Catálogos de produtos variáveis.
  • Jogos em tempo real.

Escalabilidade: O Grande Diferencial

Como crescer quando o sistema fica lento?

Vertical (SQL)

Comprar um servidor MAIOR (Mais RAM, Mais CPU).

Limite: Custo altíssimo e limite físico da máquina.

Horizontal (NoSQL)

Comprar MAIS servidores pequenos e ligá-los em Cluster.

Vantagem: Infinito e barato.

Tipo 1: Orientado a Documentos

Armazena dados em JSON (BSON). Flexível e hierárquico.

Principal Player: MongoDB

// Coleção: Usuarios { "_id": "user123", "nome": "Carlos", "enderecos": [ // Array dentro do registro! { "tipo": "casa", "rua": "Av Paulista" }, { "tipo": "trabalho", "rua": "Faria Lima" } ] }

No SQL, isso exigiria duas tabelas e um JOIN.

Tipo 2: Chave-Valor

O mais simples e rápido. É um grande dicionário (Hash Map). Geralmente em memória (RAM).

Principal Player: Redis

SET sessao:usuario:123 "{ 'logado': true, 'token': 'xyz' }" GET sessao:usuario:123 > "{ 'logado': true, 'token': 'xyz' }"

Uso ideal: Cache, Sessão de usuário, Carrinho de compras, Placar de jogos.

Outros Tipos de NoSQL

Colunar (Wide-Column)

Armazena dados em colunas, não linhas. Ótimo para ler poucas colunas de bilhões de registros.

Ex: Cassandra, HBase. (Usado pelo Facebook/Netflix).

Grafos

Focado em RELACIONAMENTOS. Nós e Arestas.

Ex: Neo4j. (Redes Sociais, Detecção de Fraude, Recomendação).

O Teorema CAP (Brewer)

Em sistemas distribuídos, escolha 2 de 3:

  • C (Consistency): Todos veem o mesmo dado ao mesmo tempo.
  • A (Availability): O sistema sempre responde (não cai).
  • P (Partition Tolerance): O sistema funciona se a rede cortar.

SQL foca em CA (Forte Consistência). NoSQL foca em AP (Alta Disponibilidade) ou CP.

Resumo: Quando usar qual?

Critério SQL (Relacional) NoSQL
Estrutura Rígida (Schema) Flexível (Schema-less)
Consistência Forte (ACID) Eventual (BASE)
Escalabilidade Vertical (Difícil) Horizontal (Fácil)
Melhor para Transações, Dados complexos interligados Big Data, Logs, Catálogos, Analytics
