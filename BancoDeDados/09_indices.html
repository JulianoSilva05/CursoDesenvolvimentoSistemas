O Problema: Full Table Scan

Imagine procurar um nome em uma lista telefônica desordenada. Você teria que ler nome por nome até achar.

Isso é o Full Table Scan: O banco lê a tabela inteira para achar um registro.

Sem Índice

Busca em 1 milhão de linhas = 1 milhão de leituras.

Tempo: 5 segundos 🐢

Com Índice

Busca binária (B-Tree).

Tempo: 0.01 segundo 🚀

O Que é um Índice?

É uma estrutura de dados auxiliar (geralmente Árvore B+) que guarda o valor da coluna e um ponteiro para onde o dado está no disco.

Funciona exatamente como o índice remissivo no final de um livro.

Nem tudo são flores...

Se índices são tão bons, por que não criar em todas as colunas?

Vantagens (Leitura)

  • SELECT muito rápido (WHERE).
  • Ordenação instantânea (ORDER BY).
  • Agrupamento rápido (GROUP BY).

Desvantagens (Escrita)

  • Ocupa espaço em disco.
  • Deixa INSERT, UPDATE e DELETE mais lentos (o banco precisa atualizar a tabela E o índice).

Tipos de Índices

Tipo Descrição Exemplo
PRIMARY O rei dos índices. Único e Não Nulo. ID, CPF
UNIQUE Garante unicidade, mas permite NULL (depende do SGBD). Email, Login
INDEX Índice comum. Acelera busca, permite repetidos. Nome, Data
FULLTEXT Busca em textos longos (como Google). Descrição, Artigo

Comandos DDL para Índices

-- Criar índice simples CREATE INDEX idx_nome_cliente ON clientes(nome); -- Criar índice composto (Melhora busca por Estado E Cidade) CREATE INDEX idx_localizacao ON enderecos(estado, cidade); -- Apagar índice (Se estiver atrapalhando a escrita) DROP INDEX idx_nome_cliente ON clientes;

Onde Criar Índices?

O segredo é a Seletividade.

Alta Seletividade (BOM)

Muitos valores diferentes.

Ex: CPF, Email, Data de Nascimento.

O índice consegue filtrar muito bem.

Baixa Seletividade (RUIM)

Poucos valores repetidos.

Ex: Sexo (M/F), Status (Ativo/Inativo).

O índice não ajuda muito, o banco prefere ler tudo.
