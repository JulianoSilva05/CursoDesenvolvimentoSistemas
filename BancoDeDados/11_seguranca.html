Segurança e Controle de Acesso

Protegendo os dados e gerenciando permissões

Curso de Desenvolvimento de Sistemas - SENAI

Duração estimada: 2 horas

Por que Segurança?

O Banco de Dados é o ativo mais valioso de uma empresa. Ele contém dados de clientes, financeiro, segredos industriais.

Pilares da Segurança da Informação (CIA):

  • Confidencialidade: Só quem tem permissão vê.
  • Integridade: O dado não é alterado indevidamente.
  • Disponibilidade: O dado está acessível quando necessário.

Gerenciamento de Usuários (DCL)

Nunca use o usuário root (admin) para conectar sua aplicação!

-- Criar um usuário CREATE USER 'vendedor'@'localhost' IDENTIFIED BY 'senha123'; -- Dar permissão (GRANT) GRANT SELECT, INSERT ON Loja.Pedidos TO 'vendedor'@'localhost'; -- Tirar permissão (REVOKE) REVOKE INSERT ON Loja.Pedidos FROM 'vendedor'@'localhost';

Princípio do Menor Privilégio

Um usuário deve ter APENAS as permissões estritamente necessárias para fazer seu trabalho. Nada mais.

  • O estagiário precisa deletar o banco? Não.
  • O sistema de relatórios precisa alterar dados? Não (só SELECT).

SQL Injection (Injeção de SQL)

É a falha de segurança mais comum em bancos de dados.

Ocorre quando o sistema concatena o texto do usuário diretamente na query.

-- Código vulnerável (PHP/Java/C#) query = "SELECT * FROM Users WHERE name = '" + userInput + "'"; -- Se o usuário digitar: ' OR '1'='1 -- A query vira: SELECT * FROM Users WHERE name = '' OR '1'='1'; -- Retorna TUDO!

Como evitar SQL Injection?

Use Prepared Statements (Consultas Preparadas).

O banco trata o input do usuário como TEXTO puro, jamais como comando.

-- Código seguro query = "SELECT * FROM Users WHERE name = ?"; execute(query, userInput);

Configurando Acesso

Cenário: Você tem um usuário 'estagiario'.

Escreva os comandos SQL para:
